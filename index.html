Fundamentals

DevOps is a cultural and technical movement that unifies software development (Dev) and IT operations (Ops) to shorten the systems development lifecycle while delivering features, fixes, and updates frequently in close alignment with business objectives.

Core Principles:

  • Collaboration: Break down silos between Dev and Ops teams
  • Automation: Automate repetitive tasks across the delivery pipeline
  • Continuous Integration/Delivery: Merge code frequently and deliver reliably
  • Monitoring & Feedback: Observe systems in production and act on data
  • Infrastructure as Code: Manage infrastructure declaratively via code

Continuous Integration (CI): Developers frequently merge code changes into a shared repository. Each merge triggers automated builds and tests to detect integration issues early.

Continuous Delivery (CD): Extends CI — every change that passes automated tests is automatically prepared and ready for release to production. A human still approves the final deployment.

Continuous Deployment: Fully automated — every change that passes the pipeline is automatically deployed to production without human intervention. Requires robust test coverage and monitoring.

Infrastructure as Code (IaC) is the practice of managing and provisioning infrastructure through machine-readable configuration files rather than manual processes.

Why it matters:

  • Version Control: Infrastructure changes are tracked just like application code
  • Reproducibility: Identical environments can be spun up reliably
  • Speed: Provision complex infrastructure in minutes
  • Consistency: Eliminates configuration drift and "works on my machine" problems
  • Documentation: Code itself serves as living documentation

Popular tools: Terraform, Pulumi, AWS CloudFormation, Ansible

Microservices is an architectural approach where an application is built as a collection of small, independently deployable services, each responsible for a specific business capability.

Relation to DevOps: Microservices and DevOps are complementary. Microservices allow individual teams to develop, deploy, and scale their services independently — aligning with DevOps's goal of frequent, reliable deployments. DevOps practices (CI/CD, containerization, observability) are essential to manage the complexity microservices introduce.

  • Each service can be deployed independently, enabling faster release cycles
  • Failures are isolated to individual services
  • Teams can choose the best technology per service

Shift Left means moving testing, security, and quality activities earlier in the software development lifecycle — to the "left" on the timeline.

Instead of testing after development is complete, teams integrate testing into every stage, from code writing to code review to CI pipeline. This catches bugs early when they're cheapest to fix.

Shift Left Security (DevSecOps): Security scanning tools run in the CI pipeline so vulnerabilities are found during development, not in production. Examples: static analysis (SAST), dependency scanning, container image scanning.

Containers & Orchestration

Docker is a platform for developing, shipping, and running applications in containers — lightweight, portable units that package code and all its dependencies.

Key differences from VMs:

  • Isolation: VMs virtualize the entire hardware including an OS kernel. Containers share the host OS kernel and isolate only the user space.
  • Size: VM images are GBs; container images are typically MBs.
  • Startup: VMs take minutes; containers start in seconds or milliseconds.
  • Overhead: VMs have heavier resource overhead; containers are much lighter.

Kubernetes (K8s) is an open-source container orchestration platform that automates the deployment, scaling, and management of containerized applications.

Problems it solves:

  • Scaling: Automatically scales applications up/down based on load
  • Self-healing: Restarts failed containers, reschedules pods on healthy nodes
  • Service Discovery: Automatically exposes and discovers services
  • Rolling Updates: Zero-downtime deployments with rollback capability
  • Resource Management: Optimally places workloads across nodes
  • Configuration Management: Manages secrets and config maps

Pod: The smallest deployable unit in Kubernetes. A Pod wraps one or more containers that share network namespace and storage. Containers in a pod communicate via localhost.

Deployment: A higher-level abstraction that manages a set of identical Pods. It handles rolling updates, rollbacks, and ensures the desired number of Pod replicas are always running.

Service: An abstraction that provides a stable network endpoint to access a set of Pods. Since Pod IPs are ephemeral, Services provide a consistent DNS name and IP. Types include ClusterIP, NodePort, LoadBalancer, and ExternalName.

  • Use official, minimal base images (e.g., alpine, distroless) to reduce attack surface and image size
  • Multi-stage builds: Separate build dependencies from runtime artifacts
  • Order layers from least to most frequently changing to maximize cache hits
  • Never run as root: Add a non-root user with USER instruction
  • Use .dockerignore to exclude unnecessary files
  • Avoid storing secrets in image layers — use environment variables or secret managers
  • Pin dependency versions for reproducible builds
  • Use HEALTHCHECK instructions

Kubernetes uses several abstractions for persistent storage:

  • PersistentVolume (PV): A piece of storage provisioned by an admin or dynamically via a StorageClass. It has a lifecycle independent of any Pod.
  • PersistentVolumeClaim (PVC): A request for storage by a user. It binds to a PV and is mounted into a Pod.
  • StorageClass: Defines the type and behavior of storage (e.g., gp2 on AWS, premium-ssd on Azure). Enables dynamic provisioning.
  • StatefulSets: Unlike Deployments, StatefulSets provide stable network identities and persistent storage for stateful applications (databases, message queues).

For cloud-native apps, CSI (Container Storage Interface) drivers allow Kubernetes to use cloud block storage, NFS, Ceph, etc.

CI/CD Pipelines

A typical CI/CD pipeline includes these stages:

  1. Source: Code commit triggers the pipeline (Git push/PR)
  2. Build: Compile code, resolve dependencies, create artifacts
  3. Test: Unit tests, integration tests, code coverage
  4. Security Scan: SAST, dependency vulnerability checks, container scanning
  5. Artifact Storage: Push built artifacts/images to a registry
  6. Deploy to Staging: Deploy to a staging/QA environment
  7. Acceptance Tests: End-to-end tests, smoke tests in staging
  8. Deploy to Production: Manual gate (CD) or fully automated (Continuous Deployment)
  9. Monitor: Observe metrics and trigger rollback if anomalies detected

GitOps is an operational framework where Git is the single source of truth for both application and infrastructure configuration. An operator (like ArgoCD or Flux) continuously reconciles the live cluster state with the desired state declared in Git.

Key differences from traditional CI/CD:

  • Pull vs Push: GitOps agents pull changes from Git rather than a CI system pushing to environments
  • Declarative: The entire desired state is declared in Git — not scripted
  • Drift Detection: GitOps continuously compares live state to desired state and auto-corrects
  • Audit Trail: Every change to production is a Git commit with a clear author

Blue-Green Deployment: Two identical production environments ("blue" and "green") exist simultaneously. Traffic is switched from blue (current) to green (new) all at once. Rollback is instant — just switch traffic back. Requires double infrastructure cost.

Canary Deployment: New version is gradually rolled out to a small subset of users (e.g., 5%), while the rest continue using the old version. If metrics look healthy, traffic is progressively shifted (5% → 25% → 100%). Allows real-world validation with minimal blast radius.

When to use which: Blue-green is ideal for predictable, all-or-nothing releases. Canary is better for risk mitigation on uncertain changes.

Secrets must never be stored in plaintext in source code or CI configs. Best practices:

  • CI-native secret stores: GitHub Actions Secrets, GitLab CI Variables — encrypted at rest, injected as environment variables
  • Dedicated secret managers: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault. Credentials are fetched dynamically at runtime with short-lived tokens.
  • Kubernetes Secrets + External Secrets Operator: Syncs secrets from external stores into K8s Secrets
  • Rotation: Rotate secrets regularly and automate rotation to reduce exposure window
  • Least privilege: Each pipeline stage should only have access to the secrets it needs

Key considerations for microservices CI/CD:

  • Monorepo or Polyrepo: With a monorepo, use path-based triggers to only build changed services. Polyrepo gives each service its own pipeline.
  • Independent pipelines per service: Each service builds, tests, and deploys independently with its own versioning
  • Contract testing: Use Pact or similar tools to verify service API contracts don't break consumers
  • Environment promotion: dev → staging → production with automated gates
  • Service mesh integration: Test traffic management, circuit breakers during pipeline
  • Observability gates: Automatically validate error rates and latency after deployment before promoting

Cloud & Infrastructure

Terraform is an open-source IaC tool by HashiCorp that allows you to define, provision, and manage cloud infrastructure using a declarative configuration language (HCL).

How it works:

  1. Write: Define resources in .tf files using HCL
  2. Plan: terraform plan previews what changes will be made without applying them
  3. Apply: terraform apply provisions/modifies the actual infrastructure
  4. State: Terraform maintains a state file (terraform.tfstate) that maps configuration to real-world resources

Terraform supports 1000+ providers: AWS, Azure, GCP, Kubernetes, GitHub, etc.

  • IaaS (Infrastructure as a Service): Provides raw compute, storage, and networking. You manage OS, runtime, middleware, and applications. Examples: AWS EC2, Azure VMs, GCP Compute Engine.
  • PaaS (Platform as a Service): Provides a managed platform including OS, runtime, and middleware. You manage only the application and data. Examples: Heroku, AWS Elastic Beanstalk, Google App Engine.
  • SaaS (Software as a Service): Fully managed applications delivered over the internet. You manage nothing infrastructure-related. Examples: Salesforce, Gmail, Slack.

DevOps primarily operates at the IaaS and PaaS layers, often building internal PaaS experiences on top of IaaS.

Local state files are dangerous in teams — they cause conflicts and loss of the source of truth. Best practices:

  • Remote backends: Store state in S3 + DynamoDB (AWS), Azure Blob Storage, or Terraform Cloud. These provide locking to prevent concurrent modifications.
  • State locking: DynamoDB tables lock state during operations so two engineers can't apply simultaneously
  • Workspaces: Use Terraform workspaces or separate state files per environment (dev/staging/prod)
  • Sensitive data: State files may contain secrets — ensure the backend is encrypted and access-controlled
  • Never commit state files to version control — add *.tfstate to .gitignore

A service mesh is a dedicated infrastructure layer (typically implemented as sidecar proxies) that handles service-to-service communication in a microservices architecture.

Capabilities:

  • Mutual TLS (mTLS) for zero-trust service authentication
  • Traffic management: load balancing, retries, timeouts, circuit breaking
  • Observability: distributed tracing, metrics, logs per service-to-service call
  • Canary/A-B traffic splitting at the network level

Popular implementations: Istio, Linkerd, Consul Connect

When to use: When you have many microservices and need consistent observability, security, and traffic control without modifying application code. Adds operational complexity — only introduce when problems justify it.

  • Multiple AZs: Deploy across at least 3 Availability Zones within each region for fault tolerance
  • Global load balancing: Route 53 with latency-based or failover routing, or AWS Global Accelerator
  • Database replication: Aurora Global Database for cross-region replication with sub-second RPO
  • Active-Active vs Active-Passive: Active-Active routes traffic to both regions simultaneously; Active-Passive keeps a standby region that activates on failover
  • Data consistency: Design for eventual consistency at global scale; use event sourcing and CQRS patterns
  • Disaster Recovery metrics: Define RTO (Recovery Time Objective) and RPO (Recovery Point Objective) — these drive architecture decisions
  • Regular runbooks and DR drills to verify failover works as expected

Monitoring & Observability

The three pillars of observability are:

  • Metrics: Numerical measurements over time (CPU usage, request rate, error rate, latency). Good for dashboards and alerting. Tools: Prometheus, Datadog, CloudWatch.
  • Logs: Timestamped records of discrete events. Invaluable for debugging specific incidents. Tools: ELK Stack, Loki, Splunk.
  • Traces: Records of a request's journey across multiple services. Shows bottlenecks and latency in distributed systems. Tools: Jaeger, Zipkin, AWS X-Ray, OpenTelemetry.

Effective observability combines all three — use metrics to detect a problem, traces to locate it, and logs to understand it.

SLI (Service Level Indicator): A quantitative measurement of a service's behavior. Examples: request latency, error rate, availability percentage, throughput.

SLO (Service Level Objective): A target value for an SLI. Example: "99.9% of requests should respond in under 200ms." SLOs are internal targets that teams aim to meet.

SLA (Service Level Agreement): A contractual agreement with customers about service levels. Typically less strict than SLOs. Violating an SLA may trigger penalties.

Error Budget: The amount of unreliability permitted within an SLO period. If you have a 99.9% SLO, your monthly error budget is ~43 minutes of downtime.

Prometheus is an open-source monitoring system that collects metrics by scraping HTTP endpoints at regular intervals.

Data Model: Every metric is a time series identified by a metric name and key-value label pairs. Example: http_requests_total{method="GET", status="200"}

Metric types:

  • Counter: Monotonically increasing (request count, errors)
  • Gauge: Can go up or down (CPU usage, memory)
  • Histogram: Samples observations in buckets (latency distribution)
  • Summary: Similar to histogram, calculates quantiles on client side

Querying is done with PromQL. Alertmanager handles notifications. Often paired with Grafana for visualization.

Distributed tracing tracks a single request as it flows through multiple microservices, recording timing and metadata at each hop. It helps identify latency bottlenecks and failure points across service boundaries.

Key concepts: A Trace represents the entire request journey. A Span is a single unit of work within a trace. Spans are linked via a trace-id propagated in request headers.

OpenTelemetry (OTel) is a vendor-neutral CNCF project that provides a unified SDK and API for collecting traces, metrics, and logs. It solves instrumentation fragmentation — you instrument once with OTel and export to any backend (Jaeger, Zipkin, Datadog, Honeycomb) without changing code.

Incident Response Process:

  1. Triage: Assess severity. Declare the incident level (P1/P2/P3). Page relevant responders.
  2. Mitigate first: Restore service as quickly as possible (rollback, feature flag off, reroute traffic). Don't investigate root cause while the service is down.
  3. Communicate: Update the status page and notify stakeholders. Keep communication frequent.
  4. Investigate: Once stable, use logs, metrics, and traces to identify root cause. Form hypotheses and validate against data.
  5. Document: Write a blameless post-mortem with timeline, root cause, contributing factors, and action items.
  6. Follow-through: Complete post-mortem action items to prevent recurrence.

Security (DevSecOps)

DevSecOps integrates security practices into every stage of the DevOps lifecycle, making security a shared responsibility rather than a bottleneck gate at the end of development.

Traditional security treats security as a separate phase — a security team reviews/scans before release. This creates friction, delays, and vulnerabilities found late are expensive to fix.

DevSecOps approach:

  • Security tools embedded in CI/CD (SAST, DAST, SCA, container scanning)
  • Developers educated on secure coding practices
  • Security requirements defined in user stories
  • Threat modeling done early in design
  • Infrastructure security validated as code via policy-as-code tools (OPA, Checkov)

Principle of Least Privilege (PoLP): Every entity (user, service, process) should have only the minimum permissions necessary to perform its function, and no more.

Application in cloud environments:

  • IAM roles with granular policies: Grant specific actions on specific resources instead of wildcards (*)
  • Service accounts per workload: Each application/service gets its own identity with scoped permissions
  • No long-lived credentials: Use IAM roles with short-lived tokens (AWS IAM Roles for EC2/EKS) instead of access keys
  • Regular access reviews: Audit and remove unused permissions
  • Workload Identity: In Kubernetes, use IRSA (AWS) or Workload Identity (GCP) to map K8s service accounts to cloud IAM roles
  • RBAC: Use Role-Based Access Control to limit what users and service accounts can do
  • Network Policies: Default-deny all traffic; explicitly allow required communication
  • Pod Security Standards: Use Restricted policies to prevent privileged containers, host namespace access
  • Image Security: Use only trusted registries; scan images for vulnerabilities in CI; use signed images
  • Secrets management: Avoid storing secrets in plain K8s Secrets (base64 is not encryption); use External Secrets Operator + Vault
  • Audit logging: Enable API server audit logs to track who did what
  • Keep K8s updated: Regularly patch the control plane and node OS
  • mTLS: Use a service mesh for encrypted service-to-service communication

Policy as Code (PaC) defines compliance, security, and operational policies as code that can be version-controlled, tested, and automatically enforced.

Open Policy Agent (OPA) is a general-purpose policy engine. Policies are written in Rego language. OPA evaluates JSON inputs against Rego policies to produce allow/deny decisions.

OPA Gatekeeper integrates OPA with Kubernetes as an admission controller. When you create/modify K8s resources, Gatekeeper evaluates them against defined constraints before allowing the operation. Example policies: "All containers must have resource limits", "Images must come from our private registry", "No privileged containers allowed".

Checkov and Terrascan serve a similar purpose for Terraform — scanning IaC before it's applied.

Zero-trust assumes no implicit trust based on network location — every request must be authenticated and authorized, regardless of whether it comes from inside or outside the network perimeter.

Implementation strategies:

  • Identity-based access: Every workload and user has a cryptographic identity; access decisions are identity-driven
  • mTLS everywhere: Enforce mutual TLS for all service-to-service communication (service mesh)
  • Micro-segmentation: Fine-grained network policies that allow only necessary communication
  • Continuous verification: Authenticate and authorize on every request, not just at session start
  • BeyondCorp / IAP: Replace VPN with context-aware access proxies that evaluate device posture, user identity, and request context

Version Control & Collaboration

A branching strategy defines how teams use branches to organize development and releases.

Common models:

  • Gitflow: Long-lived main and develop branches, with feature/release/hotfix branches. Good for versioned software, complex for CI/CD.
  • GitHub Flow: Single main branch; feature branches are short-lived and merged via PRs. Simple and works well with CI/CD.
  • Trunk-Based Development: All developers commit to a single main branch (or very short-lived branches < 1 day). Requires strong CI, feature flags. Preferred for high-velocity teams.

Modern DevOps teams often prefer Trunk-Based Development as it minimizes merge conflicts and enables true CI.

Semantic Versioning (SemVer) uses the format MAJOR.MINOR.PATCH:

  • MAJOR: Breaking/incompatible API changes
  • MINOR: New backward-compatible functionality
  • PATCH: Backward-compatible bug fixes

Why it matters: In microservices and library ecosystems, consumers need to understand the impact of updates. SemVer communicates the risk level of upgrading. CI/CD pipelines can automate version bumps based on commit messages (Conventional Commits + tools like semantic-release).

Feature flags (also called feature toggles) are configuration switches that allow you to enable or disable features at runtime without deploying new code.

Enabling continuous deployment:

  • Deploy incomplete code to production with the feature hidden behind a flag
  • Test in production with a subset of users before broad rollout
  • Instantly roll back a problematic feature by toggling the flag off — no redeployment needed
  • Decouple deployment from release — ship code continuously but control when users see features

Tools: LaunchDarkly, Unleash, Flagsmith, AWS AppConfig.

A Pull Request (PR) is a request to merge code changes from one branch to another, providing a forum for code review before integration.

Good code review practices:

  • Small, focused PRs: Easier to review; faster to merge. Aim for < 400 lines changed.
  • Clear description: Explain what the change does and why
  • Automated checks first: Lint, tests, security scans should pass before human review
  • Review for correctness, maintainability, and security — not just style
  • Constructive feedback: Blameless, focused on code not the author
  • Required reviewers: Enforce minimum approvals + CODEOWNERS for critical files

Monorepo: All projects/services in one repository. Used by Google, Meta, Microsoft.

  • ✅ Atomic cross-project changes in a single commit
  • ✅ Shared tooling and dependency management
  • ✅ Easier code reuse and discoverability
  • ❌ Requires specialized tooling (Bazel, Nx, Turborepo) to scale CI
  • ❌ Access control is harder

Polyrepo: Each service has its own repository.

  • ✅ Clear ownership and autonomy per service
  • ✅ CI only builds what changed (by default)
  • ❌ Cross-repo changes are painful
  • ❌ Duplicate tooling and configuration across repos

Scripting & Automation

Ansible is an agentless configuration management and automation tool. It connects via SSH and executes tasks defined in YAML playbooks. It's primarily used for software configuration, application deployment, and ad-hoc automation.

Terraform vs Ansible:

  • Terraform: Infrastructure provisioning (create/destroy VMs, networks, databases). Declarative, manages state, idempotent.
  • Ansible: Configuration management (install packages, configure software, deploy apps). Procedural, stateless.

They're complementary: Terraform provisions infrastructure, Ansible configures it. Modern teams increasingly use cloud-init or container images instead of Ansible for immutable infrastructure.

Idempotency: An operation that produces the same result regardless of how many times it's applied. In IaC, running terraform apply twice should result in the same infrastructure state — no unintended changes. Critical for automation safety.

Immutable Infrastructure: Instead of patching and modifying existing servers, you replace them entirely with new instances when changes are needed. Bake all configuration into an AMI or container image. Deploy the new version; retire the old.

Benefits of immutability:

  • Eliminates configuration drift
  • Rollback is as easy as deploying the previous image
  • Environments are always in a known, reproducible state

Key practices for production Bash scripts:

  • set -euo pipefail at the top — exit on errors, treat unset variables as errors, fail on pipe failures
  • Meaningful error messages: Use trap to catch errors and log context
  • Idempotent operations: Check if a state already exists before trying to create it
  • Input validation: Check required arguments and environment variables exist
  • Use shellcheck: Lint Bash scripts to catch common mistakes
  • Avoid parsing ls output — use globs instead
  • Quote all variables: "$variable" to prevent word splitting
  • Logging: Write logs to stderr; write usable output to stdout

Helm is the package manager for Kubernetes. It packages K8s manifests into charts — reusable, versioned, configurable bundles.

How it helps:

  • Templating: Uses Go templates to generate manifests dynamically from values
  • Reusability: Charts can be shared (Artifact Hub) and customized per environment
  • Release management: Tracks deployed versions; supports helm rollback
  • Dependency management: Charts can declare dependencies on other charts

Typical workflow: helm install my-app ./chart -f values-prod.yaml. Alternatives: Kustomize (overlay-based, no templating), raw manifests with environment substitution.

Configuration drift occurs when a system's actual state diverges from its intended/declared state, usually due to manual changes, failed automation, or hotfixes applied directly to production.

Detection:

  • terraform plan reveals drift between state and live infrastructure
  • GitOps operators (ArgoCD, Flux) continuously detect and report drift
  • AWS Config, Chef InSpec, or Puppet report policy violations

Prevention:

  • Immutable infrastructure: Replace rather than modify servers
  • Lock down manual changes: Enforce that all changes go through IaC pipelines
  • GitOps auto-remediation: Automatically reconcile drift back to the desired state
  • Restrict console/SSH access in production environments

Advanced Topics

Chaos Engineering is the practice of intentionally injecting failures into a system to discover weaknesses before they cause unplanned outages. The goal is to build confidence in the system's resilience.

Safe implementation:

  • Define steady state: Know what "normal" looks like before injecting chaos
  • Start small: Begin in non-production, then gradually move toward production with careful controls
  • Limit blast radius: Use feature flags to confine the experiment to a small percentage of traffic
  • Have a kill switch: Always be able to immediately stop the experiment
  • Run during business hours: When the team is available to respond

Tools: Chaos Monkey (Netflix), Litmus, Gremlin, AWS Fault Injection Simulator.

FinOps (Financial Operations) is a cultural practice that brings financial accountability to cloud spending, enabling teams to make cost-aware architectural decisions.

DevOps contributions to cost management:

  • Right-sizing: Analyze actual resource utilization and resize instances; use tools like AWS Compute Optimizer
  • Auto-scaling: Scale down during off-peak hours instead of running peak capacity 24/7
  • Spot/Preemptible instances: Use for non-critical, fault-tolerant workloads (up to 90% savings)
  • Reserved/Savings Plans: Commit to predictable workloads for discounted rates
  • Tagging strategy: Tag all resources with team/service/environment for cost attribution
  • Ephemeral environments: Spin up PR environments on demand; auto-destroy when merged

eBPF (extended Berkeley Packet Filter) allows sandboxed programs to run in the Linux kernel without modifying kernel source or loading kernel modules. It enables powerful, low-overhead observability and networking capabilities.

Use cases in DevOps:

  • Observability: Cilium Hubble and Pixie use eBPF for automatic service dependency mapping and L7 tracing without code instrumentation
  • Security: Runtime security tools (Falco, Tetragon) use eBPF to detect and prevent malicious syscalls in real time
  • Networking: Cilium replaces kube-proxy with eBPF for higher performance K8s networking
  • Performance profiling: Continuous profiling tools like Parca use eBPF for production flame graphs with minimal overhead

A postmortem (or incident review) is a structured analysis conducted after an incident to understand what happened, why it happened, and how to prevent recurrence.

Standard postmortem sections: Impact summary, timeline, root cause analysis, contributing factors, action items with owners and deadlines.

Blameless culture means focusing on system and process failures rather than individual mistakes. Key principles:

  • People made reasonable decisions given the information available at the time
  • The goal is to improve systems, not punish individuals
  • Blame suppresses incident reporting and learning
  • Complex systems fail in complex ways — usually multiple contributing factors, not one person

Pioneered by Google SRE; now a standard DevOps practice.

  • HPA (Horizontal Pod Autoscaler): Scales the number of Pod replicas based on CPU/memory utilization or custom metrics. Works by adjusting the replicas field of a Deployment based on observed metrics vs target thresholds.
  • VPA (Vertical Pod Autoscaler): Adjusts the CPU and memory requests/limits of containers based on historical usage. Can set recommendations only, or automatically update pods (requires restart).
  • Cluster Autoscaler: Scales the number of nodes in the cluster when pods can't be scheduled due to insufficient resources.
  • KEDA (Kubernetes Event-Driven Autoscaling): Scales workloads based on external event sources — Kafka consumer lag, SQS queue depth, Prometheus metrics, HTTP request rate. Can scale to zero.

CALMS is a framework that describes the core values and dimensions of a DevOps culture:

  • C — Culture: Break silos; foster collaboration, shared responsibility, and psychological safety
  • A — Automation: Automate repetitive processes — builds, tests, deployments, infrastructure
  • L — Lean: Apply lean principles — eliminate waste, reduce batch size, focus on flow
  • M — Measurement: Measure everything; use data to drive decisions and identify bottlenecks
  • S — Sharing: Share knowledge, tools, and learnings across teams; open communication

CALMS is useful for DevOps transformation assessments to identify which dimensions need the most improvement.

Database migrations in CD environments require careful coordination to avoid downtime and deployment failures.

Key principles:

  • Backward-compatible migrations: Each migration must work with both the current and next version of the application (expand-contract pattern)
  • Expand-Contract (Parallel Change):
    1. Add new column/table (expand)
    2. Deploy app code that writes to both old and new
    3. Backfill data
    4. Deploy app code that only uses new column
    5. Drop old column (contract)
  • Tools: Flyway, Liquibase for version-controlled migrations run in CI pipeline
  • Never: Rename or drop columns in the same deployment as the code change that stops using them

Container Registry: A storage system specifically for Docker/OCI container images. Images are pushed/pulled using the Docker registry protocol. Examples: Docker Hub, AWS ECR, GCR, GitHub Container Registry.

Artifact Repository: A broader storage system for binary artifacts of any type — JARs, npm packages, Python wheels, Helm charts, raw files. Examples: JFrog Artifactory, Sonatype Nexus, AWS CodeArtifact.

Key overlap: Modern artifact repositories (Artifactory, Nexus) also support container images. In practice, teams use a dedicated container registry for images and an artifact repo for language packages. Both serve as a central, versioned, immutable store for build outputs.

Requests: The amount of CPU/memory a container is guaranteed. Used by the scheduler to find a suitable node.

Limits: The maximum CPU/memory a container can use. CPU is throttled when hitting the limit; memory limit violations cause OOMKill.

QoS Classes: Kubernetes automatically assigns one of three classes:

  • Guaranteed: Every container has equal requests and limits. Highest priority; last to be evicted.
  • Burstable: At least one container has requests set, but not equal to limits.
  • BestEffort: No requests or limits set. First to be evicted under memory pressure.

Best practice: Always set requests (for scheduling) and limits (for stability). Use LimitRanges and ResourceQuotas at namespace level.

DORA metrics (from the DevOps Research and Assessment team) are four key metrics that measure software delivery performance:

  • Deployment Frequency: How often code is deployed to production. Elite teams deploy multiple times per day.
  • Lead Time for Changes: Time from code commit to running in production. Elite teams: < 1 hour.
  • Change Failure Rate: Percentage of deployments that cause a degraded service or require hotfix. Elite: < 5%.
  • Mean Time to Restore (MTTR): How long it takes to recover from a production failure. Elite: < 1 hour.

DORA research shows these metrics correlate with organizational performance. They help teams identify bottlenecks (e.g., high CFR = poor testing; high MTTR = poor observability) and set measurable improvement goals.
